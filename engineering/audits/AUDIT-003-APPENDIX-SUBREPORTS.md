## 3. DETECTION

- **Falls (production path = pure heuristic)**: impact classification at 2 Hz against base thresholds 4/8/12/20 g × activity multipliers 1.0–1.75 (`MotionManager.swift:233–249, 2422–2521`); then a 2-s timer checks post-impact stillness: warning ≥8 s, likely ≥20 s, **emergency ≥45 s** (15/30/60 s if pre-impact walking/running) (2535–2656); 90-s timeout resets (2571); 30-s impact cooldown (261). `fallEvent.isEmergency` feeds +100 score into the threat engine (`APP\ThreatDetectionManager.swift:310–344`).
  - **Layer 2 (Apple `CMFallDetectionManager`) is DEAD**: `DebugConfig.enableAppleFallDetection=false` (`DebugConfig.swift:113`); the manager itself documents that calling `isAvailable` without the `com.apple.developer.coremotion.fall-detection` entitlement crashes (`APP\AppleFallDetectionManager.swift:52–64`). FACT: the ThreatDetectionManager "Layer 2" branch (355–370) can never fire in this build.
  - **Layer 3 (FallDetector CNN, SisFall 1D-CNN, 128×6 @50 Hz, threshold 0.7 — `ML\FallDetectorMLManager.swift:20–23,120–121`) is DORMANT**: gated on `enableHighRateIMU=false` via `IMUStreamManager.isFallPathActive` (`IMUStreamManager.swift:98–100`), so ADR-001 CNN arbitration in `ThreatDetectionManager.swift:328–332` always takes the `!cnnArbitrates` branch → byte-for-byte legacy heuristic.
- **Panic word / voice on watch**: shared tap → `VoiceKeywordGate.ingest` (`APP\VoiceKeywordGate.swift:141–183`): Silero VAD v5 CoreML (512-sample frames @16 kHz, speech ≥0.5; `APP\SileroVAD.swift:31,65`) with RMS-energy fallback; KWS = legacy PanicWordDetector DS-CNN (classes silence/unknown/yes/no/**help/stop**, threshold 0.7 — `ML\PanicWordMLManager.swift:26–28`) because the preferred CreateML model isn't bundled (§1). Detection latches 4 s (`VoiceKeywordGate.swift:80`).
- **Voice→SOS gate (V338)**: a panic word escalates only when vitals are abnormal (HR elevated/high/critical or high stress); otherwise it is advisory, +6 points (`ThreatDetectionManager.swift:410–453, 1152–1160`).
- **Sound classification**: SNAudioStreamAnalyzer with scream/shout/gunshot/glass/crash detectors (`SoundAnalysisManager.swift:532–594`); stale sound evidence expired each tick after the dance-club false-SOS (V780, `ThreatDetectionManager.swift:278–284`).
- **Cloud voice (two tiers)**: (a) features-only `voice-distress-ml` ("melody"), fed by a prosody snapshot (`ThreatDetectionManager.swift:1219–1261` — note the fabricated/proxy features: `spectralFlux` derived from danger level, MFCC slot pinned at −2.0 for a deleted model); (b) raw-audio `voice-danger-stt` ("words"), **opt-in consent gated** (`ThreatDetectionManager.swift:906`), 10-s WAV relayed via phone (preferred) or direct Supabase (1060–1081). Words-over-melody veto + floor rule are pure, tested policy (`WL\VoiceEvidencePolicy.swift:29–56, 118–138`).

- **Confidence handling**: per-signal `ThreatSignal(severity, confidence)` (`ThreatDetectionManager.swift:1660–1667`); confidences are ad-hoc formulas (e.g. HR: `min(1,(hr-130)/50+0.5)` at 471) or raw uncalibrated model probabilities; the 0.374-intercept floor rule is an explicit workaround for a model that "cannot say I don't know" (`VoiceEvidencePolicy.swift:10–14`). ## 4. DECISION / ESCALATION — as actually coded

**The de-facto pipeline (FACT):** 5-s `Timer` loop `evaluateThreats()` (`ThreatDetectionManager.swift:222`) → weighted score (weights dict 111–128) → level safe/elevated(≥30)/high(≥50)/critical(≥70 or any critical signal) (1367–1383) → sustained counter → `triggerSOS()` gates → latch `shouldTriggerSOS=true` (1512–1514) → `RedesignRootView` presents a SwiftUI sheet (`APP\RedesignRootView.swift:298–313`) → **`RedesignSOSView` runs the countdown as `@State var countdown = 10` decremented by a 1-s `Timer` in the view** (`APP\RedesignSOSView.swift:41,128–150`) → on zero, `sendSOS()` → `WatchConnectivityManager.sendSOSToiPhone(...)` (183–190) → repeats every 30 s with incrementing `alertNumber` (193–211).

- **The designed FSM is not wired (FACT).** `WL\SOSStateMachine.swift` is a tested 11-state reducer with dispatch/delivered/queued/sendFailed/live states and invariants ("help is on the way only when cloud-ACKed", 104–107) — but its only app-target reference is a comment: "SOS (press & hold) is stubbed here — wired to RaxhaWatchLogic.SOSStateMachine in Phase 5" (`RedesignRootView.swift:9`). Grep confirms no `reduce(` caller outside tests. Likewise `AppStateMachine` is mirrored by an `if/else` on `@AppStorage`, not invoked (`APP\RedesignAppView.swift:5–9, 38–49`). The shipped UI therefore has **no delivered/queued/failed states**: `SOSSentView` shows "Alert #N sent" from a local counter, not any cloud ACK (`RedesignSOSView.swift:70`) — violating the very invariant the unwired FSM encodes.
- **Tick-vs-seconds defect (FACT, high confidence):** `sustainedThreatSeconds += 1` happens once per 5-s evaluation tick (1396 vs timer at 222), but is compared to `sustainedDuration: Int = 30 // Seconds` (92) and `immediateSOSDuration = 5` (99). Real wall-clock: **150 s for "30 s", 25 s for "5 s critical"**. Downstream code believes these are seconds: `getSecondsUntilSOS()` (1602–1609), `ThreatEvidenceMicPolicy` ("against the 30 s trigger threshold", `WL\ThreatEvidenceMicPolicy.swift:40–42`, `reopenSustainedThreatFloorSeconds = 5` = actually 25 s), and log lines (1398). The v1.9.76 change 2 s→5 s (220–221) silently multiplied all escalation latencies by 2.5×.
- **Gates in `triggerSOS()` (1427–1504):** GATE 0 dedup re-trigger (1435); GATE 1 **Table Mode blocks ALL auto-SOS including falls and panic words** (1441–1444); GATE 2 10-s pickup grace (1449); GATE 3 vitals-only ladder → calm-conversation suppress / check-in (60-s grace, `WL\VitalsOnlyEscalationPolicy.swift:31`) / cooldown 30 min after "I'm OK" (`vitalsCheckInAnswered`, 1304–1313).
- **Countdown persistence: NONE (FACT).** Countdown, 30-s repeat, and pending state are in-memory view `@State` + `Timer` (`RedesignSOSView.swift:44`). No persisted deadline anywhere in the watch code (`shouldBeMonitoring` is the only persisted safety flag; the offline queue persists only already-built payloads). App kill or reboot mid-countdown → SOS silently ceases to exist. Worse: `.onDisappear` fires `onResolve` → `sosHandled()` → clears the `shouldTriggerSOS` latch on **every** dismissal including system scene teardown and swipe-down (`RedesignSOSView.swift:84–95`, comment explicitly treats swipe-dismiss as cancel).
- **Dispatch & dedup (FACT):** one UUID per SOS threaded through both routes; server dedups `ON CONFLICT (id) DO NOTHING` (`APP\WatchConnectivityManager.swift:732–734, 799–801`). V330 safety floor: paired watch ALWAYS fires Route A (WCSession `sendMessage` → fallback `transferUserInfo`, which relaunches a killed iPhone app) (745–761, 845–858); Route B (direct POST to Supabase `sos` Edge Function with live vitals snapshot, `APP\CloudSyncManager.swift:525–587`) fires when standalone/selected; 2-s Route B retry only under `enableSOSParallel` which is **false** (`DebugConfig.swift:273`). Offline: critical payloads enqueue to a UserDefaults-persisted queue, max 500 (`CloudSyncManager.swift:106–111, 1700–1716, 1966–1974`).
- **Escalation FSM vs new architecture (INFERENCE):** the actual escalation "FSM" is scattered across: threat-level enum + sustained counter (ThreatDetectionManager), check-in flags (coordinator), a latch bool, a view's `@State` countdown, and an alert-repeat timer. Nothing corresponds to ACKNOWLEDGED/ESCALATING/RESOLVED; "resolution" is view dismissal.

## 6. FAILURE MODES VISIBLE IN CODE

1. **Countdown lives in a view; latch cleared on any dismissal** (§4) — `RedesignSOSView.swift:84–95`. System teardown mid-countdown = silent cancel; reboot = total loss (no persisted deadline).
2. **Tick/seconds ×5 inflation of all sustained-threat timings** (§4) — `ThreatDetectionManager.swift:222` vs `:92,99,1396`. Field-log corroboration: the café episode "countdown built 18:01→18:06" (comment `MonitoringCoordinator.swift:783–786`) matches ~150 s build + 60 s grace, not 30+60 s.
3. **Stillness vetoes can suppress a real fall**: fall monitoring cancels when variance <0.001 ("Table detected", `MotionManager.swift:2585–2593`) and GATE 1 blocks all auto-SOS while `isTableStationary` (`ThreatDetectionManager.swift:1441–1444`). The repo's own field note records table-mode cancelling a detected fall 27 s after impact (`MonitoringCoordinator.swift:853–856`). An unconscious wearer with a very still wrist is exactly the "machine-like stillness" profile. This directly violates the new architecture's "context may never suppress a life-critical event" rule (the comment claims humans always show var >0.001; UNKNOWN whether that holds for an unconscious elder — no data in repo).
4. **2 Hz accelerometer undersampling for impacts** (`MotionManager.swift:280–282`): a fall's g-spike lasts tens of ms; at 500 ms sampling the peak is usually missed (INFERENCE — physics, not tested in repo). Detection then depends on aliased residual magnitude. The 50-Hz stream that would fix this is flag-off.
5. **Dual `ThreatDetectionManager` instances (FACT)**: ContentView deliberately builds a fresh instance and warns "Do NOT use .shared" (`APP\ContentView.swift:56–70`), yet `.shared` — a dormant instance whose mic never runs and whose detection never starts — is still read by: `CloudSyncManager.swift:409` (sos_decision_metrics snapshots), `WatchConnectivityManager.swift:485` (voice correlation id), `HealthKitManager.swift:2236` (`isSOS:` in resume sync — always false, can clobber phone SOS state), `RedesignVitals.swift:60,76` (dashboard "Voice AI" mic tile — always off). Telemetry and UI read a different brain than the one deciding.
6. **Timers everywhere, none background-safe by themselves**: detection 5 s (`ThreatDetectionManager.swift:222`), fall check 2 s (`MotionManager.swift:2538`), check-in eval 15 s / banner 10 s / permission 300 s (`MonitoringCoordinator.swift:607–632`), countdown 1 s / repeat 30 s (`RedesignSOSView.swift:140,193`). They run in background **only because** the HKWorkoutSession keeps the process alive (INFERENCE from watchOS semantics; corroborated by the code's own admission that table-sleep timers stop when ERS expires — `RaxhaApp.swift:147–149`).
7. **Silent degradation by config**: `enableAppleFallDetection=false`, `enableHighRateIMU=false`, `enableSOSParallel=false`, `enableContextualDistress` split by build flavor (`DebugConfig.swift:73,92–94,113,273`) — three advertised detection layers are actually one layer at runtime.
8. **No TODO/FIXME/HACK anywhere** (grep: zero matches) — but dozens of "Vnnn" post-mortem comments (V330 dropped SOS, V493 re-trigger loop, V696 stuck mic, V701 nav-stack storm, V760 discarded rescue score, V766 café false alarm, V780 14-min stale scream, V788 club false alarm, V867 Telugu STT garbage) — FACT: this codebase's history is an incident log embedded in comments; each fix is a point patch on the same fused scorer.
9. **Force-unwraps**: none material in the watch safety path (only dictionary-literal unwraps of compile-time keys, e.g. `MotionManager.swift:2428–2441`, and phone-side/test `try!`).

## 7. HIDDEN ASSUMPTIONS

- **Workout write permission granted** for live 5-s HR cadence (observer-only degraded mode exists, `HealthKitManager.swift:1135–1155`); HealthKit revocation only *inferred* after 15 min of silence (`MonitoringCoordinator.swift:1317–1325`).
- **BAR fires** (watchOS may skip it — acknowledged `RaxhaApp.swift:141`); **probation requires foreground** (root-cause note 273–277) — wake-from-table correctness depends on OS scheduling goodwill.
- **Phone reachable for Supabase token refresh** (§5) — standalone Route B decays silently.
- **Locale ≈ spoken language** for STT provider routing (`ThreatDetectionManager.swift:1174–1187`); the V867 note admits an en-US locale routed a Telugu emergency to the wrong provider whose "confident garbage transcript … also blocked the S1 escalation" (981–990).
- **User answers check-ins**; unconscious wearer costs one 60-s grace window (analyzed and accepted, `VitalsOnlyEscalationPolicy.swift:20–23`) — but stacked on the ×5 tick bug the real pre-countdown latency for vitals-only is ~150 s + 60 s + 10 s countdown ≈ 3.7 min (INFERENCE).
- **Human wrists always jitter** (fall/table veto §6.3). **Charging detectable only by polling** (no watchOS notification, `MotionManager.swift:304–311`). **One mic graph** (second capture would break SoundAnalysis, `MonitoringCoordinator.swift:973–976`). **Two CMMotionManagers coexist** (unvalidated risk, `IMUStreamManager.swift:25–32`). **Cloud models' probability semantics stable** (0.374 intercept hardcoded, `VoiceEvidencePolicy.swift:29`).
